二、核心设计原则（v0.2.0新增，后续迭代必须遵循）
================================================================================

【1. 案由插件化设计】
- 特殊案由不单独建顶层文件夹
- 通过 00-通用基础设施/案由适配索引.md + 各阶段 特殊案由适配/ 模板实现差异化
- 一审四阶段骨架保持统一，降低维护成本
- 5种适配案由：代位权、撤销权、人格否认、股东出资、执行异议之诉

【2. 程序分层架构】
- 一审程序是核心骨架
- 二审程序复用一审四阶段框架，核心差异在内容（上诉状vs起诉状）
- 执行程序自建轻量模块，执行异议之诉通过"套壳"复用一审框架
- 00-通用基础设施 跨程序共享

【3. 套壳方法】
- 执行异议之诉本质是一审诉讼，直接复用 01-一审程序/ 四阶段骨架
- 仅调整关键参数：当事人地位、管辖、起诉期限、前置程序、诉讼请求
- 中央套壳指引（00-套壳指引.md）统一说明套壳方法
- 各子类型README/SKILL引用中央指引

【4. 期限优先规则】
- 法院指定的期限 > 法定期限
- AI必须读取法院来文中的特定期限
- 关键期限自动提取并生成《关键期限跟踪表.md》
- 期限到达前3日提醒用户

【5. 客户指引前置】
- 客户指引放在操作文件夹内（如 01-原始案件材料/材料收集指引.md）
- 不单独建 00- 客户指引文件夹
- 指引采用勾选框格式，降低使用门槛

【6. 特殊情况说明机制】
- 01-原始案件材料/ 中预置 特殊情况说明.md 模板
- 用户填写后，AI据此检索专业材料清单，生成定制化清单
- 特殊主体和特殊证据形式在模板中勾选标注

【7. 质量检查分离原则】
- 通用质量检查清单只含通用项目
- 案由-specific检查项放在各案由适配模板中
- 避免兼容性风险损害债权人权益实现

【8. 互斥规则显式声明】
- 案外人执行异议之诉 与 第三人撤销之诉 互斥（民诉法解释第301条）
- 执行和解后的恢复执行 与 起诉 互斥（执行和解规定第9、13条）
- 上诉 与 申请执行 互斥（一审判决后二选一）
- AI必须在相关节点显式提醒互斥规则

================================================================================

【设计原则】
- 直接复用一审四阶段骨架
- 核心差异：内容适配二审场景

- 04-收官：二审判决后的路径选择（再审/执行/结案）

【关键期限】
- 上诉期限：判决15日 / 裁定10日
- 二审答辩期：收到上诉状副本后15日
- 再审申请期限：判决生效后6个月

五、执行程序（04-执行程序/） v0.2.0 新建
================================================================================

【10-执行实施主线】
- 申请执行→财产调查→财产控制→财产处置→执行完毕/终本
- 关键期限：申请执行2年、查封不动产3年/动产2年/存款1年
- 辅助措施：悬赏执行、限制消费、纳入失信名单
- 执行和解协议（风险提示：不具有强制执行力）

【20-执行审查程序】
- 五种审查程序：执行行为异议、案外人异议、追加/变更异议、执行复议、执行监督
- 关键区分：行为异议→复议；案外人异议→异议之诉
- 期限：异议审查15日、复议申请10日、异议之诉起诉15日

【30-执行异议之诉】
- 四种类型：案外人/申请执行人/追加变更/分配方案
- 套壳一审四阶段框架
- 中央套壳指引统一说明套壳方法
- 各子类型独立README+SKILL

【40-执行衍生与特殊救济】
- 执行回转：执行依据被撤销后的返还
- 参与分配：多债权人对同一被执行人（公民/其他组织）
- 执行协调：不同法院执行冲突
- 执行救助：被执行人无财产，申请人生活困难

七、案由适配详细规则
================================================================================

【代位权纠纷】
- 当事人：原告=债权人，被告=次债务人，第三人=债务人
- 管辖：次债务人住所地（被告住所地）
- 诉讼请求：必须明确"代位范围以债权人的债权为限"
- 特殊证据：向债务人发出催告的书面证明
- 质量检查：债务人未进入破产程序

【撤销权纠纷】
- 当事人：原告=债权人，共同被告=债务人和相对人（民法典合同编通则解释第44条）
- 管辖：债务人或相对人住所地
- 诉讼请求：必须同时包含"撤销"+"恢复原状/返还财产"
- 特殊证据：债权成立时间早于处分行为
- 质量检查：除斥期间（1年/5年）、"恢复财产"请求已包含

【人格否认（损害公司债权人利益责任纠纷）】
- 当事人：三种情形（债权已确认另行起诉/一并起诉/直接起诉需追加公司）
- 管辖：侵权行为地或被告住所地（不适用公司诉讼特殊管辖）
- 诉讼请求：股东对公司债务承担连带责任
- 特殊证据：人格混同证据（财务账册、人员交叉、业务混同）
- 2025年案由变更：原"股东损害债权人利益/实际控制人损害债权人利益"四级案由已删除，统一归入三级案由"307.损害公司债权人利益责任纠纷"

【股东出资纠纷】
- 当事人：原告=债权人或公司，被告=股东
- 管辖：存在争议（公司住所地或被告住所地）
- 诉讼请求：在未出资/抽逃出资范围内承担补充赔偿责任
- 特殊证据：公司章程、出资凭证、终本裁定（加速到期）、抽逃路径链条
- 注意区分：股东出资纠纷 vs. 损害公司债权人利益责任纠纷

【执行异议之诉】
- 四种类型，当事人地位各异
- 管辖：执行法院专属管辖
- 前置程序：必须先经过执行异议程序
- 起诉期限：收到裁定之日起15日
- 本质：一审诉讼，套壳一审框架

八、关键期限汇总
================================================================================

【一审程序】
- 答辩期：15日（法院指定优先）
- 举证期限：法院指定（通常≥30日）
- 上诉期：判决15日 / 裁定10日

- 上诉期：判决15日 / 裁定10日
- 二审答辩期：15日
- 再审申请期：判决生效后6个月

【执行程序】
- 申请执行期：2年
- 执行立案：7日
- 执行通知：10日
- 查封不动产：3年；动产：2年；存款：1年
- 执行异议审查：15日
- 执行异议之诉起诉：收到裁定后15日
- 执行复议申请：收到裁定后10日
- 执行和解恢复执行：和解协议约定履行期届满后2年

【通用规则】
- 法院指定的期限 > 法定期限
- 期限到达前3日提醒用户

十、已知问题与迭代备忘（v0.2.2）
================================================================================

【严重】上下文压缩导致目录结构修复不彻底
- 现象：在批量创建/重命名目录、更新README目录树、替换路径引用的过程中，
  由于单次对话涉及文件数量过多（>150个），AI的上下文被压缩后，部分已修复
  的内容又被覆盖回旧状态，或新增修复遗漏了之前已修复的位置。
- 具体表现：
  ① README.md 中的目录树与实际目录反复出现不一致；
  ② 执行程序模块编号（10/20/30/40→01/02/03/04）的替换在部分文件标题中遗漏；
  ③ 同一目录下的子目录编号冲突在多次修复中此消彼长。
- 根因：上下文压缩导致AI无法同时记住所有已修改文件的状态，每次新操作
  只能基于压缩后的摘要执行，丢失了精确的文件级记忆。
- 应对措施（v0.2.3已落实）：
  ① ✅ 后续目录结构调整必须一个阶段一个阶段过，不要一次性操作多个程序；
  ② ✅ 每次修改后立即用脚本验证（重复编号检查、路径引用检查）；
     → 验证脚本：`.kimi/scripts/validate_project.py`

十一、防上下文压缩工具链（v0.2.3 新增）
================================================================================

【问题背景】
项目文件总数超过150个，单次对话涉及大量文件读写时，AI上下文会被压缩，
导致：已修复内容被覆盖回旧状态、新增修复遗漏之前已修复的位置、目录树
反复不一致等问题。

【解决方案：四件工具 + 一套工作流】

工具1：验证脚本
- 路径：`.kimi/scripts/validate_project.py`
- 功能：自动检查8大项（目录编号重复、README目录树一致性、顶层README完整性、
  阶段文件夹SKILL/README配对、版本号一致性、占位README、无效路径引用、
  防压缩工具完整性）
- 用法：`python .kimi/scripts/validate_project.py`

工具2：一键运行封装
- 路径：`.kimi/scripts/run-validation.ps1`
- 功能：PowerShell封装，一键运行验证并显示结果
- 用法：`.\.kimi\scripts\run-validation.ps1`

工具3：修改清单模板
- 路径：`.kimi/templates/modification-checklist.md`
- 功能：每次修改前创建实体化任务清单，记录目标、范围、检查项、基线、
  验证结果、遗留问题。不依赖AI记忆。

工具4：防压缩工作流指引
- 路径：`.kimi/workflows/anti-context-compression.md`
- 功能：定义"一次对话一个模块、外部记忆优先、验证驱动"的核心原则，
  提供标准五步工作流（创建清单→导出基线→执行修改→运行验证→更新清单）。

【工作流口诀】
一单一块一验证，清单基线不依赖。
- 一单：一次只创建一个修改清单
- 一块：一次只处理一个模块/程序
- 一验证：每次修改后立即验证
- 清单：把进度写在文件里
- 基线：修改前导出快照
- 不依赖：不依赖AI记忆，依赖文件系统

【应用效果】
- 验证脚本首次运行发现：0个严重问题、35个警告（34个占位README+1个README树差异）、
  1个信息。无效路径引用已修复1处。
- 目录编号重复问题已通过v0.2.2修复，验证脚本确认无复发。

十二、AI 协作教训记录（v0.2.3 新增）
================================================================================

【教训1】PowerShell 中文编码问题（2026-04-22）
- 现象：多次用 python -c 传递多行字符串，因 PowerShell 默认 GBK 编码失败
- 根因：PowerShell 默认代码页 936，python -c 多行字符串在 PowerShell 中会被
  错误解析（转义+编码双重问题）
- 规避：复杂逻辑绝不使用 python -c；优先用 WriteFile 创建 .py 文件再执行；
  Shell 仅用于运行已有脚本或简单命令
- 文件记录：`.kimi/lessons-learned.md`

【教训2】目录不存在导致 WriteFile 失败（2026-04-22）
- 规避：WriteFile 前先确保父目录存在（New-Item -Force 或 os.makedirs）

【教训3】过度依赖 Shell 的 python -c（2026-04-22）
- 规避：超过2行的逻辑，直接写 .py 文件再执行
