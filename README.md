## 这是什么

`民事诉讼律师.md` 是一个**人与 AI 协作的诉讼工作框架**。它将民事诉讼的全流程拆解为**四大程序**（一审程序、二审程序、再审程序、执行程序），每个程序内按四阶段骨架组织（或轻量模块），每个阶段配备：

- **README.md** — 给使用者看的操作说明
- **SKILL.md** — 给 AI 看的工作流指引（Kimi / ChatGPT / Claude 均可识别）
- **参考模板** — 22 份可直接复用的 Markdown 模板
- **阶段产出区** — AI 生成文书的标准输出位置

使用者按阶段放入案件材料、唤起 AI、审核 AI 产出，即可完成从立案到结案的全部文书准备工作。涉及提交法院的正式文书，建议由具备法律执业资格的专业人士审核把关。

## 📄 为什么使用 Markdown 格式

本项目所有模板和产出文件均采用 **Markdown（`.md`）** 格式，这是一种轻量级标记语言，具有以下优势：

- **纯文本，永不过时**：不像 Word 或 PDF 依赖特定软件，Markdown 是纯文本，任何设备、任何系统都能打开
- **结构清晰，AI 友好**：标题、列表、表格等结构对 AI 来说非常容易识别和解析，这让 AI 能更准确地理解你的工作流和产出内容
- **版本控制友好**：配合 Git 使用，可以清晰追踪每一份文书的修改历史
- **专注内容，不纠结排版**：用简单的符号就能表达层级和重点，不会被字体、颜色分散注意力
- **一键转换**：可轻松导出为 Word、PDF、HTML 等任何格式

### 推荐的 Markdown 阅读器

| 工具 | 平台 | 特点 |
|------|------|------|
| **Typora** | Win/Mac/Linux | 所见即所得，支持导出 Word/PDF，最推荐 |
| **Obsidian** | Win/Mac/Linux/iOS/Android | 双向链接、本地优先，适合管理多个案件 |
| **VS Code** + Markdown 插件 | Win/Mac/Linux | 免费开源，配合 Git 使用最佳 |
| **MarkText** | Win/Mac/Linux | 开源免费，类似 Typora |
| **系统自带记事本** | 任何平台 | 无需安装，直接打开 `.md` 文件即可阅读 |

> 💡 **需要 Word 格式？** 你可以在以下任一环节将 Markdown 转换为 Word：
> 1. **Typora / Obsidian**：直接导出 `.docx`，格式保留完整
> 2. **Pandoc**（命令行工具）：`pandoc 起诉状.md -o 起诉状.docx`
> 3. **在线转换工具**：如 [md2pdf.netlify.app](https://md2pdf.netlify.app/) 等
> 4. **手动复制**：直接在 Markdown 阅读器中全选复制，粘贴到 Word 中即可
>
> 提交法院的正式文书，建议转换为 Word 后按法院要求的字体（通常宋体小四/四号）和格式调整。

## 快速开始

> 📖 **首次使用？请先看《使用指南.md》** — 包含 Kimi Code CLI 安装配置、三种使用方式（命令行/网页版/通用AI工具）、各阶段提示语示例、常见问题等完整教程。

### 环境准备

1. **下载本项目**到本地工作目录
2. **准备 AI 工具**：推荐使用 Kimi（本项目 SKILL.md 按 Kimi 的 skill 格式编写），ChatGPT / Claude 亦可使用
3. **准备一个空文件夹**作为案件工作区，将本项目复制进去

### 一审程序：阶段一（战略把脉）

```bash
# 1. 进入阶段一
cd 01-一审程序/01-战略把脉/

# 2. 将当事人提供的原始材料放入 01-原始案件材料/
#    如为特殊案由，同时填写 01-原始案件材料/特殊情况说明.md

# 3. 唤起 AI，上传 SKILL.md，说：
#    "请帮我整理这个案件的材料，完成阶段一工作流"

# 4. AI 自动生成 03-阶段产出/ 中的全部文件
# 5. 律师审核 AI 产出，补充类案检索、调整策略方案
```

### 一审程序：阶段二（诉讼材料及来文管理）

```bash
# 1. 将阶段一产出复制到阶段二
#    cp -r 01-一审程序/01-战略把脉/03-阶段产出/* 01-一审程序/02-诉讼材料及来文管理/01-上一阶段产出/
#    （Windows: xcopy "01-一审程序\01-战略把脉\03-阶段产出\*" "01-一审程序\02-诉讼材料及来文管理\01-上一阶段产出\" /E /I）

# 2. 让当事人填写 02-客户情况说明/诉讼意图情况说明.md

# 3. 唤起 AI，上传 SKILL.md，说：
#    "请基于阶段一产出和客户情况说明，完成阶段二工作流"

# 4. AI 自动生成起诉状/答辩状、证据目录、证据说明等
#    如为特殊案由，自动调用案由适配模板
# 5. 专业人士审核修改后，打印/盖章/提交法院
```

### 一审程序：阶段三（庭审交锋）



| 时点一 | 开完庭后 | 《代理词》 | — |
| 时点二 | 调解/和解成功 | 调解书/和解协议 | 直接进入时点五（结案复盘） |
| 时点三 | 收到判决书后 | 《上诉状》/《强制执行申请书》 | 上诉→02-二审程序 / 执行→04-执行程序 |
| 时点四 | 判决生效后 | 《退费申请书》 | — |
| 时点五 | 案件终局后 | 《结案报告》+《复盘总结》 | — |

### 二审程序 — 独立四阶段工作流

二审程序配备独立完整的四阶段工作流，核心特点在于：
- **阶段一**：分析一审判决的错误点，评估改判可能性
- **阶段二**：起草上诉状/二审答辩状（不是起诉状/答辩状）
- **阶段三**：二审庭审准备（审理范围限于上诉请求）
- **阶段四**：二审判决后的处理（再审/执行/结案）

### 再审程序 — 特殊救济，三阶段

再审程序分为三个阶段：
- **阶段一（再审申请）**：评估是否符合民诉法第 211 条的再审事由，撰写再审申请书
- **阶段二（再审审理）**：法院裁定再审后，按原审程序重新审理，配备独立完整的四阶段工作流
- **阶段三（收官）**：分析再审裁判效力，选择后续路径（上诉/检察监督/执行回转/结案）

> ⚠️ 再审不是必然发生的程序。大多数再审申请被驳回。只有符合法定再审事由且在判决生效后 6 个月内申请的，才有可能进入再审审理。

### 执行程序 — 四大模块

| 模块 | 定位 | 核心内容 |
|------|------|---------|
| 01-执行实施主线 | 执行的核心 | 申请执行→财产查控→处置→完毕/终本 |
| 02-执行审查程序 | 执行中的救济 | 行为异议/案外人异议/复议/监督 |
| 03-执行异议之诉 | 实体争议的诉讼 | 四种异议之诉，独立诉讼类型，配备独立完整的四阶段工作流 |
| 04-执行衍生与特殊救济 | 特殊情形 | 执行回转/参与分配/协调/救助 |

## 设计原则

1. **案由插件化**：特殊案由（代位权/撤销权/人格否认/股东出资/执行异议之诉）不单独建顶层文件夹，而是通过 `00-通用基础设施/案由适配索引.md` + 各阶段 `特殊案由适配/` 模板实现差异化。保持四阶段骨架统一，降低维护成本。
2. **程序分层**：四个程序均配备**独立完整的工作流**，可根据案件情况独立执行。各程序之间通过程序转换索引衔接。
3. **原始材料保护**：AI 不移动、不修改、不删除原始文件，只复制整理。
4. **阶段产出放最后**：每个阶段的产出文件夹编号最大，便于直接识别和复制。
5. **双版本设计**：关键文书区分"内部工作版"（含策略预判、证明力评估）和"提交法院版"（正式、客观、便于复制）。
6. **AI 能力边界**：AI 生成的是**初稿**，须经律师审核后方可提交法院；AI 严禁编造案例；AI 无法访问付费法律数据库的完整内容。
7. **卷宗索引不搬运**：阶段三、四通过相对路径直接引用前一阶段卷宗，保持单份文件、避免冗余。
8. **原告 / 被告双模式**：阶段二内置角色判定逻辑，自动切换"建设型"（原告）和"拆弹型"（被告）工作流。
9. **期限优先规则**：法院指定的期限优先于法定期限。AI 必须读取法院来文中的特定期限，而非仅依赖法定基准。
