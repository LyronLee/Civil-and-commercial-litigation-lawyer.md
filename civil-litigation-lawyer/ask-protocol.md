# 自动询问协议（Ask Protocol）

> 本文件定义agent在信息缺失时的自动询问机制。
> 核心原则：渐进式披露、推断优先、选项引导、紧急提醒。

---

## 信息三层模型

### L1 必须信息（缺失则无法开始）

| 序号 | 信息项 | 判定依据 | 询问方式 |
|------|--------|---------|---------|
| 1.1 | 案件基本类型 | 无法从材料推断 | 选项式询问 |
| 1.2 | 我方角色 | 无法从材料推断 | 选项式询问 |
| 1.3 | 当前案件状态 | 无法从卷宗识别 | 选项式询问 |
| 1.4 | 是否有法院文书 | 卷宗中无法院文书 | 确认式询问 |
| 1.5 | 争议标的金额 | 无法从材料推断 | 直接询问 |
| 1.6 | 对方当事人信息 | 材料中无明确被告 | 直接询问 |

### L2 关键信息（影响策略方向，可边做边补）

| 序号 | 信息项 | 影响 | 处理方式 |
|------|--------|------|---------|
| 2.1 | 管辖法院/合同履行地 | 管辖选择 | 阶段一中补充 |
| 2.2 | 财产线索 | 保全/执行策略 | 阶段二中补充 |
| 2.3 | 是否已委托律师 | 工作方式调整 | 自然询问 |
| 2.4 | 关键期限（上诉期/执行期等） | 紧急程度 | 自动扫描+提醒 |
| 2.5 | 特殊案由迹象 | 适配规则调用 | 阶段0.5识别 |
| 2.6 | 关联案件 | 策略影响 | 材料扫描+询问 |

### L3 可选信息（有则更好，没有也能推进）

| 序号 | 信息项 | 用途 |
|------|--------|------|
| 3.1 | 当事人背景（企业/个人） | 策略参考 |
| 3.2 | 历史合作/纠纷关系 | 事实梳理 |
| 3.3 | 对方履约能力 | 执行预判 |
| 3.4 | 调解意愿 | 策略选择 |

## Ask模式执行流程

### 触发条件

以下任一情况触发Ask模式：
1. L1信息缺失 ≥ 1项
2. 用户仅描述案情，未提供任何材料
3. 卷宗材料无法识别案件状态
4. 用户主动要求"帮我梳理一下需要准备什么"

### 执行步骤

```
Step 1: 评估缺失信息
    对照L1清单，标记缺失项
    按优先级排序（紧急期限相关优先）

Step 2: 首轮询问（1-3个问题）
    每次最多提出3个结构化问题
    优先询问最高优先级的缺失信息

Step 3: 接收回答并更新
    提取回答中的关键信息
    更新信息状态表
    尝试推断其他信息（推断优先原则）

Step 4: 循环检查
    L1仍缺失？→ 继续询问（回到Step 2）
    L1已完整？→ 退出Ask模式，进入正常工作流
```

### 退出条件

- L1信息全部收集完毕
- 用户明确表示"信息就这些，先开始吧"
- 用户表示"其他信息我后续补充"

> 退出Ask模式时，生成信息收集摘要，标注"以下信息待后续补充"。

## 结构化提问模板

### 模板1：案件类型识别

```
为了确定适用的法律框架，请告诉我这是什么类型的纠纷：

A. 合同纠纷（买卖、借款、租赁、建设工程、服务等）
B. 侵权纠纷（人身损害、财产损害、知识产权侵权等）
C. 物权纠纷（所有权确认、返还原物、排除妨碍等）
D. 婚姻家庭纠纷（离婚、抚养、继承、分家析产等）
E. 劳动争议
F. 公司/股权纠纷
G. 其他：______

您可以直接回复选项字母，也可以详细描述案情。
```

### 模板2：角色确认

```
在这个案件中，您是：

A. 原告（主动起诉对方）
B. 被告（被对方起诉，需要应诉）
C. 第三人（被追加参加诉讼）
D. 申请执行人（胜诉后申请强制执行）
E. 被执行人（被申请强制执行）
F. 还不确定

如果是A/D，我将按"建设型"工作流推进；
如果是B/E，我将按"拆弹型"工作流推进。
```

### 模板3：案件状态确认

```
目前案件进展到什么阶段了？

A. 刚决定起诉/应诉，还没立案
B. 已经向法院提交了起诉状/答辩状
C. 收到了法院传票，准备开庭
D. 已经开完庭，等判决
E. 已经收到一审判决书
F. 已经收到二审判决书
G. 判决已生效，对方不履行
H. 正在执行中
I. 其他：______
```

### 模板4：对方当事人信息

```
请提供对方当事人的基本信息：

- 对方是自然人还是企业/组织？
- 姓名/名称：
- 住所地/注册地址：
- 联系方式（如有）：

这些信息将用于确定管辖法院和填写起诉状/答辩状。
```

### 模板5：争议金额确认

```
本案涉及的争议金额大约是多少？

- 标的金额：______ 元
- 是否涉及财产保全需求？（是/否/不确定）

这将影响诉讼费计算、管辖级别和保全策略。
```

### 模板6：紧急期限提醒

```
⚠️ 紧急提醒：检测到关键期限临近！

根据您提供的[判决书/裁定书/传票]，以下期限需要注意：

- [期限类型]：截止日期 [日期]（剩余 [X] 天）
- [期限类型]：截止日期 [日期]（剩余 [X] 天）

建议优先处理与上述期限相关的任务。
是否立即开始？
```

## 推断优先规则

agent应尽可能从已有信息中推断，减少询问次数：

| 已知信息 | 可推断信息 | 推断逻辑 |
|---------|-----------|---------|
| "对方欠我货款" | 案件类型=合同纠纷 | "货款"→买卖合同 |
| "我是买方，对方没交货" | 角色=原告 | 主张对方违约→原告 |
| "收到法院传票了" | 状态=已立案，角色=被告 | 传票→已立案；收到传票→被告 |
| "合同签在深圳" | 管辖=深圳法院 | 合同签订地→管辖连接点 |
| "有律师函" | 已有前期沟通 | 律师函→已尝试协商 |
| "2019年的事" | 注意诉讼时效 | 超过3年→需核查时效中断 |

> 推断结果应在后续对话中向用户确认："根据您的描述，我推断本案是[XX纠纷]，您是[原告/被告]，是否正确？"

## 渐进式披露原则

### 每轮对话最多3个问题

❌ 错误示范：
```
请告诉我：1.案件类型 2.您的角色 3.当前状态 4.对方信息 5.争议金额 6.管辖法院 7.是否有证据...
```

✅ 正确示范：
```
为了开始分析，我需要先确认三个基本情况：

1. 这是什么类型的纠纷？（合同/侵权/物权/婚姻家庭/其他）
2. 您是原告还是被告？
3. 目前案件到什么阶段了？（还没立案/已立案/已开庭/已判决）
```

### 根据回答动态调整后续问题

```
用户回答："合同纠纷，我是原告，还没立案"

→ 推断：建设型工作流，一审阶段一
→ 下一批问题：
  1. 对方是企业还是个人？名称/姓名是什么？
  2. 合同履行地/签订地在哪里？
  3. 争议金额大约多少？
```

## 信息收集摘要格式

退出Ask模式时，生成以下摘要：

```markdown
## 信息收集摘要

### 已确认信息
- 案件类型：[XX纠纷]
- 角色：[原告/被告]
- 当前状态：[阶段]
- 对方当事人：[名称/姓名]
- 争议金额：[金额]

### 待补充信息（L2）
- [信息项]：将在[阶段X]中补充
- [信息项]：建议用户后续提供

### 自动推断信息（待确认）
- [推断项]：基于[依据]推断，请用户确认
```

## 特殊场景处理

### 场景1：用户说"我也不知道该准备什么"

→ 进入"全引导模式"：
1. 先问案件类型和基本事实
2. 根据类型展示"通常需要准备的材料清单"
3. 询问用户手头有哪些材料
4. 标记缺失材料及获取方式

### 场景2：用户提供的材料与描述不符

→ 友好提醒：
"您提到本案是[XX纠纷]，但我从材料中看到[YY内容]，是否我理解有误？请确认案件类型。"

### 场景3：用户拒绝回答某些问题

→ 尊重用户选择：
"了解，您暂时不想提供[XX信息]。我会基于现有信息先开始工作，该信息在[阶段X]时再补充也不影响整体进度。"

### 场景4：多被告/多原告案件

→ 逐一确认：
"本案涉及多个当事人，请逐一确认：
- 第一被告：[信息]
- 第二被告：[信息]
- ..."
