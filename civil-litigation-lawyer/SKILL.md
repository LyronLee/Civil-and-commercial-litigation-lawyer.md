# 民事诉讼律师 · 单Skill工作框架

> **核心理念**：用户只需说"我有一个案子"，agent自动完成其余一切。
> 
> 本skill将民事诉讼全流程内化，无需复制物理项目结构，无需手动选择阶段，
> 无需上传多个SKILL.md。所有模板、规则、工作流均内置于skill中，
> agent根据案件状态自动路由、动态生成。

---

## 触发条件

用户表达以下意图时触发本skill：

### 通用触发
- "我有一个案子" / "帮我处理这个案件"
- "起诉" / "应诉" / "上诉" / "申请执行" / "申请再审"
- 提供卷宗文件夹路径
- 上传案件材料并请求处理
- 任何与民事诉讼相关的法律咨询或文书需求

### 二审程序触发
- "我要上诉"
- "对一审判决不服"
- "收到对方上诉状了"
- "帮我写上诉状"
- "二审答辩状怎么写"
- "评估一下改判可能性"

### 再审程序触发
- "我要申请再审"
- "对生效判决不服，想再审"
- "看看能不能再审"
- "再审申请书怎么写"
- "收到驳回再审申请裁定，下一步怎么办"

### 执行程序触发
- "申请强制执行"
- "帮我写强制执行申请书"
- "执行案件怎么推进"
- "被执行人没钱怎么办"
- "执行和解协议怎么写"
- "申请限制消费/纳入失信"
- "对执行行为有异议"
- "我是案外人，对执行标的有异议"
- "不服追加被执行人的裁定"
- "对异议裁定不服"
- "执行程序违法，怎么救济"
- "执行依据被撤销了，要回转"

- "被执行人没钱，我生活困难，能申请救助吗"

## 核心工作流

### Step 0：信息收集与案件状态识别

#### 0.1 接收用户输入

**情况A：用户提供了卷宗文件夹路径**
- 读取该文件夹及其子文件夹中的所有文件
- 按文件类型分类统计（PDF/图片/Word/Excel/文本）
- 识别文件名中的关键信息（合同、协议、判决书、传票、律师函等）

**情况B：用户未提供路径，仅描述案情**
- 进入 **Ask模式**（详见 ask-protocol.md）
- 按L1信息清单逐项收集

**情况C：用户提供了部分材料（如上传几个文件）**
- 分析已提供材料
- 对照L1清单，识别缺失信息
- 对缺失项进入Ask模式

#### 0.2 案件状态自动识别

基于卷宗内容分析，判断当前案件状态：

| 识别信号 | 推断状态 | 进入程序 |
|---------|---------|---------|
| 无法院文书，有合同/协议/沟通记录 | 新案/准备起诉 | 一审程序 · 阶段一 |
| 有起诉状/答辩状，无传票 | 已立案/等待开庭 | 一审程序 · 阶段二 |
| 有传票/举证通知书 | 已收到开庭通知 | 一审程序 · 阶段三 |
| 有判决书/裁定书（一审） | 一审已裁判 | 一审程序 · 阶段四 |
| 有上诉状/二审传票 | 二审进行中 | 二审程序 |
| 有再审申请书/裁定 | 再审程序 | 再审程序 |
| 有生效法律文书，申请执行 | 执行程序 | 执行程序 |
| 有执行通知书/财产查控记录 | 执行进行中 | 执行程序 |

> ⚠️ **如无法自动识别**：询问用户"请问案件目前进展到什么阶段了？"

#### 0.3 紧急期限扫描（并行执行）

在读取卷宗的同时，扫描所有日期信息：
- 发现判决书/裁定书 → 提取日期，计算上诉期/再审申请期
- 发现传票 → 提取开庭日期，计算准备时间
- 发现执行通知书 → 提取履行期限
- 发现律师函/沟通记录中的日期 → 计算诉讼时效

**如任何期限在7日内到期**：
- 立即向用户发出紧急提醒
- 优先处理与该期限相关的任务
- 在plan.md中标注"紧急期限"

#### 0.4 信息完整性评估

对照 ask-protocol.md 中的 L1/L2 清单评估：
- L1完整 → 进入 Step 1
- L1缺失 → 进入Ask模式，收集完毕后进入 Step 1

### Step 1：程序路由与角色判定

#### 1.1 程序选择

根据Step 0识别的案件状态，自动选择程序：

```
案件状态
    │
    ├─ 新案/准备起诉 ─────────────────────→ 01-一审程序
    │
    ├─ 已收到一审判决 ──────────────────→ 02-二审程序 或 04-执行程序
    │    （不服判决→上诉；服判但对方不履行→执行）
    │
    ├─ 判决已生效，认为有错 ────────────→ 03-再审程序
    │    （再审申请期限：判决生效后6个月，除斥期间）
    │
    ├─ 胜诉后对方不履行 ────────────────→ 04-执行程序
    │    （申请执行期限：2年，适用中止、中断）
    │
    ├─ 执行中遇到异议 ──────────────────→ 04-执行程序/执行审查 或 执行异议之诉
    │    （行为异议→复议；标的异议→异议之诉；追加异议→异议之诉）
    │
    ├─ 执行依据被撤销 ──────────────────→ 04-执行衍生与特殊救济/执行回转
    │
    ├─ 多个债权人执行同一被执行人 ────────→ 04-执行衍生与特殊救济/参与分配
    │
    ├─ 不同法院执行冲突 ────────────────→ 04-执行衍生与特殊救济/执行协调
    │
    ├─ 执行不能且生活困难 ──────────────→ 04-执行衍生与特殊救济/执行救助
    │
    ├─ 二审发回重审 ────────────────────→ 01-一审程序（重新审理，另行组成合议庭）
    │
    ├─ 再审发回重审 ────────────────────→ 01-一审程序（按一审程序重新审理）
    │    或 02-二审程序（按二审程序重新审理）
    │
    ├─ 再审改判/维持 ───────────────────→ 04-执行程序（新裁判执行或执行回转）
    │    或 检察监督路径（再审维持原判→向检察院申请监督）
    │
    └─ 驳回再审申请 ────────────────────→ 检察监督路径（向检察院申请检察建议或抗诉）
```

#### 1.2 角色判定

| 角色 | 判定依据 | 工作流特点 |
|------|---------|-----------|
| 原告/上诉人/申请执行人 | 主动发起诉讼/上诉/执行 | 建设型：收集材料→制定策略→起草文书→主动推进 |
| 被告/被上诉人/被执行人 | 被动应诉/被申请执行 | 拆弹型：分析对方材料→寻找漏洞→起草答辩状→防守反击 |
| 第三人 | 被追加或主动申请参加 | 辅助型：明确立场→准备陈述意见 |
| 案外人 | 对执行标的主张权利 | 异议型：证明实体权利→排除强制执行 |

> 角色判定后，后续所有阶段自动切换对应视角。

#### 1.3 案由识别与适配

基于材料内容判断案由：

**普通案由**（直接执行标准流程）：
- 合同纠纷、侵权纠纷、物权纠纷、婚姻家庭纠纷、继承纠纷等

**特殊案由**（自动调用适配规则，详见 rules.md）：
- 债权人代位权纠纷
- 债权人撤销权纠纷
- 损害公司债权人利益责任纠纷（人格否认）
- 股东出资纠纷
- 执行异议之诉（4个子类型）

> 识别为特殊案由后，自动应用 rules.md 中的适配规则，调整当事人地位、管辖、诉讼请求等。

- 当前阶段：[阶段X]

- 预计产出：[文件清单]

### Step 3：分阶段执行

用户确认计划后，按阶段执行。每个阶段：

#### 3.1 读取本阶段所需材料
- 从卷宗文件夹中读取相关文件
- 读取前一阶段的产出（如存在）
- 读取通用规则（rules.md中的相关部分）

#### 3.2 执行阶段工作流
- 按 content.md 中对应程序/阶段的详细指令执行
- 动态调用 templates.md 中的模板结构生成文书
- 应用 rules.md 中的案由适配规则（如适用）

#### 3.3 生成阶段产出
- 所有产出文件直接写入卷宗文件夹
- 命名规范：`[序号]-[文件名称].md`
- 每个文件头部标注："初稿，需律师审核后使用"

#### 3.4 阶段质量检查
- 按 quality-check.md 中的清单逐项检查
- 不通过则修正
- 通过后向用户汇报阶段完成

#### 3.5 阶段衔接
- 询问用户："本阶段已完成，是否进入下一阶段？"
- 用户确认后，自动读取本阶段产出，进入下一阶段
- 用户可要求修改本阶段产出后再进入下一阶段

---

### Step 4：质量检查与最终交付

#### 4.1 通用质量检查（所有产出）

| 序号 | 检查项 | 合格标准 |
|------|--------|---------|
| 1 | 当事人信息 | 完整、准确（姓名/名称、住所、联系方式） |
| 2 | 管辖法院 | 正确（级别管辖+地域管辖） |
| 3 | 诉讼请求 | 明确、具体、可执行 |
| 4 | 事实与理由 | 围绕诉讼请求展开，有证据支撑 |
| 5 | 证据材料 | 真实、合法、与待证事实有关联 |
| 6 | 法条引用 | 现行有效，条文内容准确 |
| 7 | 期限计算 | 准确，考虑送达时间 |
| 8 | 文件编码 | UTF-8，中文显示正常 |
| 9 | AI产出标注 | 所有提交法院的文书标注"初稿，需律师审核后使用" |

#### 4.2 程序特定检查

详见各程序的 content.md 中的"质量检查"部分。

#### 4.3 交付说明

向用户说明：
1. 所有产出文件的位置（卷宗文件夹中的路径）
2. 需要审核的关键点
3. 下一步建议
4. 关键期限提醒

## 关键设计原则

1. **极简用户操作**：用户只需提供卷宗路径和基本案情，其余由agent自动完成
2. **主动询问**：信息缺失时自动进入Ask模式，不等待用户主动提供
3. **动态模板**：不依赖物理模板文件，所有文书按结构规范动态生成
4. **自动路由**：根据案件状态自动选择程序/阶段，用户只需确认
5. **期限优先**：任何期限临近时自动提醒并优先处理
6. **质量内建**：每个阶段结束时自动质量检查，不通过不进入下一阶段
7. **单卷宗工作区**：所有产出写入用户指定的卷宗文件夹，不复制项目结构
8. **专业审核提醒**：所有产出标注"初稿，需律师审核后使用"
9. **严禁编造案例**：AI检索案例必须标明来源URL，未检索到案例时如实说明，不得虚构填充

## AI能力边界与禁止事项

### 类案检索（严格限制）

**必须执行的操作**：
1. 生成类案检索报告前，**必须**调用法律数据库（华宇元典法律数据库 yuandian_law）进行案例检索
2. 优先检索以下专业法律数据库：
   - **人民法院案例库**（rmfyalk.court.gov.cn）— 指导性案例、参考案例
   - **中国裁判文书网**（wenshu.court.gov.cn）— 裁判文书全文
   - **北大法宝**（pkulaw.com）— 法律法规、案例、期刊
   - **法信**（faxin.cn）— 法律条文、案例、观点
   - **国家法律法规数据库**（flk.npc.gov.cn）— 现行有效法律法规
3. 记录检索执行记录：平台、关键词、时间、结果数量
4. 将检索到的真实案例填入报告，标注可核实的来源URL

**法条引用（强制核实）**：
1. 引用法条前，**必须**调用法律数据库（华宇元典法律数据库 yuandian_law）检索核实：
   - 法条是否现行有效（是否被修订、废止）
   - 条文内容是否准确（与官方文本一致）
   - 条号是否正确（注意2023年《民事诉讼法》修正后的新条号）
2. 优先核实来源：
   - **国家法律法规数据库**（flk.npc.gov.cn）
   - **北大法宝**（pkulaw.com）
   - **中国人大网**（npc.gov.cn）
3. 如检索发现法条已修订/废止，必须引用现行有效版本，并标注修订情况
4. 所有法条引用必须标注"经[数据库名称]核实"

**严禁执行的操作**：
1. ❌ **严禁编造案例**：不得基于AI训练数据虚构案号、法院名称、裁判要旨
2. ❌ **严禁虚构来源**：不得填写虚假URL或不可核实的来源
3. ❌ **严禁推断伪装**：不得将"推断的典型案例类型"伪装为真实检索结果
4. ❌ **严禁填充编造**：未检索到案例时，不得在报告中虚构案例填充
5. ❌ **严禁引用失效法条**：不得引用已废止或修订前的旧法条
6. ❌ **严禁凭记忆引用法条**：所有法条引用必须经过数据库检索核实

**未检索到案例时的正确处理**：
```
## 四、检索结果
经在[平台名称]以关键词"[关键词]"进行检索，未找到与本案争议焦点直接相关的已公开裁判案例。

## 五、替代分析
虽无直接类案，但可参考以下法律适用思路：
1. [法律原则分析]
2. [学理观点引用]
```

### 其他AI能力边界

1. **法条引用**：AI引用的法条必须经过专业法律数据库检索核实，确保现行有效、条文准确
2. **案例检索**：AI无法保证检索到所有相关案例，无法访问付费专业数据库完整内容
3. **案件预测**：AI不能预测案件结果，仅能做概率分析和风险提示
4. **专业判断**：AI不能替代律师的专业判断，所有策略建议需律师审核
